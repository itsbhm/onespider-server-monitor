# 🧹 Database Factory Reset Guide

This guide explains how to perform a **Hard Reset** on your OneSpider Monitor database. 

## When should you do this?
- You have finished testing the system locally and want to wipe all dummy data before going live.
- The `uptime_logs` table has corrupted or invalid logs.
- You accidentally messed up a PostgreSQL view, table, or policy.

A Hard Reset is **100% safe**. It will automatically `DROP` all existing monitor tables and views, and recreate them cleanly from scratch without affecting the rest of your Supabase account.

---

## 🛠️ Step-by-Step Instructions

1. **Navigate to Supabase**
   - Go to your [Supabase Dashboard](https://supabase.com/dashboard/projects).
   - Select your monitor project.
   - On the left sidebar, click on the **SQL Editor** (the `</>` icon).

2. **Execute the Reset**
   - Click on **New Query**.
   - Clear any old code in the editor, and **Paste** the SQL code provided below.
   - Click the green **Run** button at the bottom right (or press `Ctrl+Enter`).

```sql
-- Drop if exists (safe for re-runs)
DROP VIEW IF EXISTS quarterly_uptime_report CASCADE;
DROP VIEW IF EXISTS monthly_uptime_report CASCADE;
DROP VIEW IF EXISTS weekly_uptime_report CASCADE;
DROP VIEW IF EXISTS daily_uptime_report CASCADE;
DROP TABLE IF EXISTS uptime_logs CASCADE;

-- Core table: stores every health check result
CREATE TABLE uptime_logs (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    created_at  TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    icmp_up     BOOLEAN NOT NULL,
    http_up     BOOLEAN NOT NULL,
    latency_ms  FLOAT DEFAULT 0,
    error_message TEXT
);

-- Fast lookup: latest logs first
CREATE INDEX idx_logs_created_desc ON uptime_logs (created_at DESC);

-- Fast lookup: find downtime incidents
CREATE INDEX idx_logs_downtime ON uptime_logs (created_at DESC) 
    WHERE icmp_up = FALSE AND http_up = FALSE;

-- Enable RLS (Supabase best practice)
ALTER TABLE uptime_logs ENABLE ROW LEVEL SECURITY;

-- 1. Allow service_role full access (for Node.js script insertions)
CREATE POLICY "Allow service role full access" ON uptime_logs
    FOR ALL USING (true) WITH CHECK (true);

-- 2. Allow PUBLIC (anon) read-only access (for public Status Page dashboard)
CREATE POLICY "Allow public read access" ON uptime_logs
    FOR SELECT USING (true);

-- 3. Grant table permissions to anon role
GRANT SELECT ON uptime_logs TO anon;

-- ─── Daily Report ───────────────────────────────
CREATE OR REPLACE VIEW daily_uptime_report AS
SELECT 
    date_trunc('day', created_at AT TIME ZONE 'Asia/Kolkata') AS report_date,
    COUNT(*) AS total_checks,
    ROUND((SUM(CASE WHEN icmp_up = TRUE OR http_up = TRUE THEN 1 ELSE 0 END)::numeric / COUNT(*)::numeric) * 100, 2) AS uptime_percentage,
    ROUND(AVG(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS avg_latency_ms,
    ROUND(MIN(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS min_latency_ms,
    ROUND(MAX(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS max_latency_ms,
    SUM(CASE WHEN NOT icmp_up AND NOT http_up THEN 1 ELSE 0 END) AS total_downtime_checks
FROM uptime_logs GROUP BY 1 ORDER BY 1 DESC;

-- ─── Weekly Report ──────────────────────────────
CREATE OR REPLACE VIEW weekly_uptime_report AS
SELECT 
    date_trunc('week', created_at AT TIME ZONE 'Asia/Kolkata') AS report_week,
    COUNT(*) AS total_checks,
    ROUND((SUM(CASE WHEN icmp_up = TRUE OR http_up = TRUE THEN 1 ELSE 0 END)::numeric / COUNT(*)::numeric) * 100, 2) AS uptime_percentage,
    ROUND(AVG(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS avg_latency_ms,
    ROUND(MIN(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS min_latency_ms,
    ROUND(MAX(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS max_latency_ms,
    SUM(CASE WHEN NOT icmp_up AND NOT http_up THEN 1 ELSE 0 END) AS total_downtime_checks
FROM uptime_logs GROUP BY 1 ORDER BY 1 DESC;

-- ─── Monthly Report ─────────────────────────────
CREATE OR REPLACE VIEW monthly_uptime_report AS
SELECT 
    date_trunc('month', created_at AT TIME ZONE 'Asia/Kolkata') AS report_month,
    COUNT(*) AS total_checks,
    ROUND((SUM(CASE WHEN icmp_up = TRUE OR http_up = TRUE THEN 1 ELSE 0 END)::numeric / COUNT(*)::numeric) * 100, 2) AS uptime_percentage,
    ROUND(AVG(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS avg_latency_ms,
    ROUND(MIN(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS min_latency_ms,
    ROUND(MAX(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS max_latency_ms,
    SUM(CASE WHEN NOT icmp_up AND NOT http_up THEN 1 ELSE 0 END) AS total_downtime_checks
FROM uptime_logs GROUP BY 1 ORDER BY 1 DESC;

-- ─── Quarterly Report ───────────────────────────
CREATE OR REPLACE VIEW quarterly_uptime_report AS
SELECT 
    date_trunc('quarter', created_at AT TIME ZONE 'Asia/Kolkata') AS report_quarter,
    COUNT(*) AS total_checks,
    ROUND((SUM(CASE WHEN icmp_up = TRUE OR http_up = TRUE THEN 1 ELSE 0 END)::numeric / COUNT(*)::numeric) * 100, 2) AS uptime_percentage,
    ROUND(AVG(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS avg_latency_ms,
    ROUND(MIN(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS min_latency_ms,
    ROUND(MAX(latency_ms) FILTER (WHERE icmp_up = TRUE)::numeric, 2) AS max_latency_ms,
    SUM(CASE WHEN NOT icmp_up AND NOT http_up THEN 1 ELSE 0 END) AS total_downtime_checks
FROM uptime_logs GROUP BY 1 ORDER BY 1 DESC;

-- Grant access to views
GRANT SELECT ON daily_uptime_report TO anon;
GRANT SELECT ON weekly_uptime_report TO anon;
GRANT SELECT ON monthly_uptime_report TO anon;
GRANT SELECT ON quarterly_uptime_report TO anon;
```

🎉 **Done!** You should see a "Success, no rows returned" message. Your database is now completely clean, empty, and ready to ingest fresh data from GitHub Actions!
